# 상위 1% 클로드코드 사용자는 프롬프트 대신 '루프'를 씁니다 — Claude Code `/loop` + LOOP.md 반복 업무 운영 가이드

![Claude Code loop](https://img.shields.io/badge/claude--code-/loop-FF6D5A?style=for-the-badge&logo=Claude)

매번 똑같은 프롬프트를 복붙해서 클로드코드에 일을 시키고 계신가요? 손에 익을수록 그 반복 요청 자체가 또 하나의 일처럼 느껴지죠. 이 가이드는 프롬프트를 매번 치는 대신, Claude Code의 `/loop`과 `LOOP.md` 한 장으로 **반복 업무가 알아서 다음 단계로 굴러가는 운영 구조**를 비개발자도 그대로 따라 만들 수 있게 정리한 문서입니다.

고객 문의 분류, 리드 후속 메일 초안처럼 매번 같은 작업을 반복하는 분이라면, `LOOP.md`에 루프를 등록해두고 **하나의 프로젝트 세션에서 루프 두 개를 굴리는 전 과정**을 따라 하실 수 있습니다.

> ⚠️ **먼저 알아두세요.** `/loop`은 **내 PC에서 세션이 켜져 있는 동안, 최대 7일까지** 도는 기능입니다. 컴퓨터를 끄면 멈출 수 있고, 24시간 알아서 출근하는 직원형 자동화가 아닙니다.

## 반복 프롬프트 vs 루프: 무엇이 다른가

같은 일을 매번 새로 시키는 대신, 직원에게 "이 일감 더미를 계속 다음 칸으로 밀어둬"라고 **한 번만 지시해두는 것** — 그게 루프입니다.

![반복 프롬프트 vs 루프](./01-prompt-vs-loop.png)

| 구분 | 매번 프롬프트 | 루프 (`/loop`) |
|---|---|---|
| 누가 시작시키나 | 사람이 매번 같은 요청 입력 | 한 번 설계해두면 루프가 자동으로 |
| 사람이 하는 일 | 반복 요청 + 확인 | **설계와 확인만** |

> 💡 Claude Code를 만든 개발자도 최근 공개적으로, 이제 같은 프롬프트를 매번 치는 대신 '루프'를 걸어둔다는 취지의 이야기를 했습니다. 이렇게 일하는 방식을 '루프 엔지니어링'이라고 부르기도 합니다. 

## `/loop` · `/goal` · Hermes cron 구분

요즘 비슷한 기능이 많아 헷갈립니다. 딱 세 가지만 구분하면 됩니다.

![/loop /goal Hermes cron 구분](./02-loop-goal-cron.png)

| 기능 | 한 줄 정의 | 다음 일이 시작되는 시점 | 얼마나 오래 사나 |
|---|---|---|---|
| **`/loop`** (오늘 주인공) | 같은 작업을 간격마다 다시 실행 | 정한 시간 간격(또는 1분~1시간 자동) | 내 PC·세션 기준, **최대 7일** |
| **`/goal`** | 끝나는 조건이 될 때까지 계속 | 직전 작업이 끝나면 바로 | 조건 달성 시 자동 종료 |
| **Hermes cron** / claude Cloud routines | 정해진 시간에 일하고 보고하는 AI 직원 | 정해진 스케줄(영속) | 내 PC·세션과 무관하게 계속 |

> 📌 `/loop`은 "매일 아침 뭐 좀 확인해줘" 하는 단순 알림용이 아닙니다. 진짜 힘은 **진행 중인 일을 계속 이어받아 다음 단계로 보내는 데** 있어서, 시트에 새 행이 계속 쌓이거나 문의·리드가 끊임없이 들어오는 업무에 잘 맞습니다.

> 버전 참고: `/loop`은 Claude Code **v2.1.72** 이상, `/goal`은 **v2.1.139** 이상에서 사용 가능합니다. `claude --version`으로 확인하세요.

## 사전 준비물

| # | 준비물 | 내용 |
|---|---|---|
| ① | Claude Code 최신 버전 | `/loop`이 들어온 비교적 최근 버전. `claude --version`으로 확인 후 낮으면 업데이트 |
| ② | 작업용 프로젝트 폴더 | 운영 파일과 결과가 한 폴더에 모이도록 폴더 하나 |
| ③ | gws (googleworkspace-cli) | 구글 시트·Gmail을 다루는 **구글 워크스페이스 CLI** |
| ④ | 샘플 데이터 + 회사 맥락 문서 | 고객 피드백·아웃바운드 리드 시트 2장 + `business-context.md`·`product.md` (전부 **가상 데이터**) |

버전 확인:

```bash
claude --version
```

```text
  설치해서 시트와 지메일 연동되게 해줘.
  https://github.com/googleworkspace/cli
```

## 실습 — LOOP.md 기준으로 한 세션에서 루프 두 개 굴리기

핵심은 **관리 문서(마크다운)와 실제 결과물(시트·Gmail)을 섞지 않는 것**입니다.

![LOOP.md → 프로젝트 세션 → 시트/Gmail](./03-loopmd-session-results.png)

| 영역 | 무엇 | 예시 |
|---|---|---|
| 관리 문서 (마크다운 3장) | 사람이 읽는 운영 문서 | `CLAUDE.md`(작업 규칙) · `LOOP.md`(루프 장부) · 주간 리포트(실행 이력) |
| 하나의 프로젝트 세션 | 등록된 루프를 같은 세션에서 실행 | `feedback-triage`(30분) · `lead-followup`(1시간) |
| 실제 결과물 | 내가 실제로 일하는 곳 | 구글 시트(분류·답변초안) · Gmail 초안(발송대기) |

### 1단계 — 프로젝트 폴더 만들기

```text
mkdir loop-sample && cd loop-sample
claude
```

### 2단계 — 운영 파일부터 만들기 (CLAUDE.md · LOOP.md · 주간 리포트)

업무 루프를 걸기 전에, 제일 먼저 Claude한테 운영 파일부터 만들게 시킵니다. 아래 프롬프트를 그대로 붙여넣으면 됩니다.

```text
이 폴더를 'AI 루프 운영' 프로젝트로 세팅해줘. 아래 세 파일을 만들어줘.

1) CLAUDE.md — 이 프로젝트의 작업 규칙:
   - 모든 루프는 LOOP.md에 등록된 것만 실행한다 (멋대로 새 일 시작 금지)
   - 결과물은 구글 시트와 Gmail '초안'에만 쓴다. 메일 발송·행 삭제 등 되돌릴 수 없는 일은 사람 승인 후에만.
   - 회사·제품 맥락은 business-context.md, product.md를 참고한다.
   - 루프는 매 실행 후 이번 주 주간 리포트 파일에 한 줄 기록한다.
   - 매주 그 주에 해당하는 weekly-loop-report.md를 새로 생성한다.
2) LOOP.md — 루프 장부. loop_name, 목적, 대상, 주기, 결과 위치, 상태, 그리고 각 루프의 '실행 프롬프트 원문'을 저장한다.
3) W25-weekly-loop-report.md — 이번 주 실행 이력 표(빈 양식). (오늘은 2026-W25)
```

생성된 `LOOP.md`는 대략 이런 모양입니다.

```text
# LOOP.md — AI 루프 장부

## 등록된 루프
| loop_name       | 목적                             | 대상            | 주기 | 결과 위치             | 상태   |
|-----------------|----------------------------------|-----------------|------|-----------------------|--------|
| feedback-triage | 새 문의 분류·답변초안·FAQ후보     | 고객피드백 시트 | 30m  | 고객피드백 시트       | active |
| lead-followup   | 후속 필요 리드의 후속 메일 초안   | 아웃바운드 리드 | 1h   | 리드 시트 + Gmail 초안 | active |

## 실행 프롬프트 원문
### feedback-triage
(여기에 /loop 프롬프트 전체를 그대로 저장 — 7일 만료 후 재시작용)
### lead-followup
(여기에 /loop 프롬프트 전체를 그대로 저장)

## 규칙
- 여기 등록된 루프만 실행 (멋대로 새 일 금지)
- 메일 발송·삭제 등 되돌릴 수 없는 일은 사람 승인 후에만 (루프는 '초안'까지)
```

> 📌 세션 이름을 따로 나눠 관리하는 게 아니라, **`LOOP.md`가 실행 기준서**입니다. 각 루프마다 **프롬프트 원문을 통째로 저장**해두는 이유는, 7일 만료 후 손쉽게 재활용하기 위해서입니다.

### 3단계 — 첫 번째 루프 정의: feedback-triage (고객 문의 분류·답변초안)

`LOOP.md`의 `feedback-triage` 항목에 들어가는 실행 프롬프트입니다. **첫 줄은 반드시 간격과 지시가 같이 있는 `/loop <간격> <지시>` 형태**입니다.

```text
/loop 30m 고객피드백 시트에서 상태가 '신규'인 행을 위에서 한 건만 골라, 아래 규칙대로 처리해.
1) business-context.md·product.md를 참고해 문의요약을 분류(환불/결제/버그/기능/요금제/기타)
2) 우리 톤으로 답변초안 작성해서 답변초안 칸에 넣기
3) 자주 나올 질문이면 FAQ후보=Y 표시
4) 상태를 '검토대기'로 바꾸고 사람확인 칸은 비워두기
5) 이 루프를 LOOP.md에 loop_name=feedback-triage 로 active 등록하고, 이 프롬프트 원문을 '실행 프롬프트 원문' 섹션에 저장
6) 처리 후 W25-weekly-loop-report.md에 실행 1건 기록
gws로 해당 행만 업데이트하고, 한 번에 한 건만 처리해.
```

### 4단계 — 두 번째 루프 정의: lead-followup (리드 후속 메일 초안 → Gmail 초안)

```text
/loop 1h 아웃바운드 리드 시트에서 상태가 '후속 필요'인 행을 한 건 골라, 아래 규칙대로 처리해.
1) business-context.md·product.md를 참고해 회사명·업종·관심신호에 맞춘 후속 메일 '초안' 작성
2) gws로 Gmail 초안(draft)만 생성 — 절대 발송하지 말 것
3) 생성된 초안ID를 시트 Gmail초안ID 칸에 기록, 상태를 '발송대기'로
4) 사람확인 칸은 비워둬 (사람이 Gmail에서 직접 검토 후 발송)
5) 이 루프를 LOOP.md에 loop_name=lead-followup 로 active 등록하고, 이 프롬프트 원문을 '실행 프롬프트 원문' 섹션에 저장
6) 처리 후 W25-weekly-loop-report.md에 실행 1건 기록
```

> ⚠️ Gmail은 **초안(draft)까지만** 만들고 절대 자동 발송하지 않습니다. 사람이 Gmail에서 열어 손보고 **보내기 버튼은 직접** 누릅니다.

### 5단계 — 한 프로젝트 세션에서 등록된 루프 실행

루프별로 세션을 따로 띄우지 않습니다. 이미 열어둔 Claude Code 세션 하나에 이렇게 요청합니다.

```text
LOOP.md를 읽고, 상태가 active인 루프들을 이 프로젝트 세션에서 실행해줘.
각 루프는 LOOP.md의 loop_name, 주기, 목적, 실행 프롬프트 원문을 그대로 사용해서 /loop로 등록해.
현재 등록할 루프 목록을 먼저 요약하고, 필요한 파일이나 시트 연결이 없으면 실행하지 말고 멈춰서 알려줘.
메일 발송·행 삭제는 금지하고, Gmail은 초안까지만 만들어.
각 실행 후 W25-weekly-loop-report.md에 실행 1건을 기록해.
```

> 💡 사람은 루프를 하나씩 외워서 실행하지 않습니다. **`LOOP.md`를 최신 상태로 관리하고 "여기 적힌 루프들 실행해줘"라고 요청**하면 됩니다.

### 고급 응용 — LOOP.md가 “다음 작업”까지 고르게 하기

오늘 가이드는 실패율을 낮추기 위해 `고객피드백 처리`, `리드 메일 작성`처럼 작업 대상과 결과 위치가 명확한 루프부터 보여줍니다. 익숙해지면 `LOOP.md`를 단순한 루프 목록이 아니라 **다음 작업 판단 기준서**처럼 넓혀 쓸 수도 있습니다.

```text
/loop 1h LOOP.md와 현재 프로젝트 파일, 시트 상태를 읽고 지금 AI가 처리해도 안전한 다음 작업 1개를 골라 처리해.

선택 기준:
1) 마감 임박 > 고객 영향 > 매출 영향 순으로 우선순위를 판단한다.
2) 한 번에 하나의 작업만 선택한다.
3) 외부 발송, 삭제, 결제, 가격 확정은 절대 하지 않는다.

완료 기준:
4) 선택한 작업의 결과물이 실제 위치에 남아 있어야 한다.
   - 시트 작업이면 상태값이 `검토대기` 또는 `발송대기`로 바뀐다.
   - 문서 작업이면 초안 파일 경로가 생성된다.
   - 메일 작업이면 Gmail 초안까지만 만들고 초안ID를 기록한다.

검증 방법:
5) 처리 후 결과 위치를 다시 읽어 상태값/파일/초안ID가 있는지 확인한다.
6) 확인 실패 시 한 번만 수정 재시도하고, 그래도 실패하면 멈춘다.

멈춤 조건:
7) 사람 승인, 외부 발송, 삭제, 결제, 가격 결정이 필요하면 즉시 멈추고 확인 요청한다.
8) 한 번의 실행에서는 작업 1개만 처리하고 종료한다.

기록:
9) weekly-loop-report.md에 선택 이유, 완료 기준, 검증 결과, 사람 확인 필요 항목을 기록한다.
10) LOOP.md의 마지막 실행 시각과 상태를 업데이트한다.
```

이런 general loop는 “다음 작업을 골라줘”보다 **무엇이 완료인지, 어떻게 확인할지, 언제 멈출지**를 같이 정해두는 게 더 중요합니다.

| 설계 질문 | 꼭 넣어야 하는 기준 | 예시 |
|---|---|---|
| 무엇이 완료인가? | 루프가 스스로 확인할 수 있는 완료 조건 | 시트 상태가 `검토대기`로 바뀜 / 초안 파일이 생성됨 / 테스트가 통과함 |
| 어떻게 확인할 것인가? | 결과 검증 방법 | 시트 값 확인 / 파일 존재 확인 / 브라우저 스크린샷 확인 / 체크리스트 점수화 |
| 언제 멈출 것인가? | 하드 스톱과 사람 확인 지점 | 한 번에 1건만 처리 / 3회 실패하면 중단 / 발송·삭제·가격 결정 전 멈춤 |
| 어디에 기록할 것인가? | 다음 실행 때 이어받을 로그 위치 | `weekly-loop-report.md` / `LOOP.md` 상태값 / 시트의 처리상태·비고 칸 |

> 📌 general loop일수록 자유도를 넓히는 것보다, **완료 기준과 검증 방법을 더 좁고 명확하게 주는 것**이 중요합니다. 루프는 100% 완벽한 결과를 보장하는 장치가 아니라, 사람이 매번 하던 피드백·검증 과정을 일부 대신해 첫 결과의 품질을 끌어올리는 장치에 가깝습니다.

> ⚠️ 처음부터 이렇게 넓게 열어두면 AI가 엉뚱한 일을 고를 수 있습니다. 처음에는 오늘처럼 **특정 시트의 특정 상태 행 처리**처럼 좁게 시작하고, 익숙해진 뒤 우선순위 기준과 금지선을 추가해 조금씩 general loop로 확장하는 것을 추천합니다.

## 루프 1회가 도는 6단계

| 단계 | 하는 일 |
|---|---|
| ① 이어받기 | 시트에서 다음 '신규/후속필요' 1건 집기 |
| ② 처리 | 분류·초안 작성 |
| ③ 점검 | 우리 톤·형식 확인 |
| ④ 기록 | gws로 시트 업데이트 / Gmail 초안 생성 |
| ⑤ 상태 전진 | '검토대기'/'발송대기'로 |
| ⑥ 잠들기 | 다음 차례까지 |

> 💡 사람은 ④(기록) 이후 **결과만 검토**합니다. 끝나면 다음 차례까지 잠들었다 다시 ①로 돌아갑니다.

### 일주일 뒤 — LOOP.md로 루프 되살리기

`/loop`은 최대 7일이면 자동으로 끝납니다. 다음 주가 되면 다시 같은 폴더에서 세션을 열고 이렇게 요청하면 됩니다.

```text
LOOP.md를 읽고, 상태가 active인 루프들을 다시 시작해줘.
각 루프는 LOOP.md의 '실행 프롬프트 원문' 섹션에 저장된 프롬프트를 그대로 사용해줘.
loop_name, 주기, 목적, 안전 규칙이 빠진 루프가 있으면 실행하지 말고 먼저 알려줘.
```

> 📌 매주 똑같은 걸 다시 설계할 필요가 없습니다. 핵심은 세션 이름이 아니라 **`LOOP.md`에 남아 있는 루프 이름과 실행 프롬프트 원문**입니다.

## 한계와 주의점

솔직히 이게 만능은 아닙니다. 꼭 지켜야 할 주의점이 있습니다.

![/loop 주의점 4가지](./04-loop-safety.png)

| # | 주의점 | 내용 |
|---|---|---|
| ① | 내 PC·세션 + 최대 7일 | 살아 있는 세션에서만 돌고, 반복 루프는 7일 뒤 자동 종료 |
| ② | PC 끄면 멈춤 | 종료·재부팅하면 멈출 수 있음 — `LOOP.md`를 읽어 다시 시작 |
| ③ | 한도(quota) 소비 | 루프가 반복 실행될 때마다 구독 사용량 소비 — 처음엔 2~3개부터 |
| ④ | 보내기는 사람 | 루프는 답변·메일 '초안'까지만, 발송·완료 확정은 사람이 |

> ⚠️ 24시간 알아서 출근해 일하는 영속·정기 운영이 필요하다면, 그건 Hermes Agent의 cron 같은 직원형 도구의 영역입니다. `/loop`은 **"내가 작업하는 동안 일을 계속 밀어주는"** 추진기로 이해하시면 정확합니다.

## 자주 묻는 질문

### Q1. `/loop`은 컴퓨터를 꺼도 도나요?

아니요. `/loop`은 내 PC에서 세션이 켜져 있는 동안만 돌고, 반복 루프는 최대 7일까지입니다. 끄면 멈출 수 있고, 다시 켜면 `LOOP.md`를 읽어 active 루프를 다시 시작하면 됩니다.

### Q2. 루프가 메일을 자동으로 보내나요?

아니요. Gmail은 **'초안(draft)'까지만** 만듭니다. 보내기 버튼은 항상 사람이 직접 확인하고 누릅니다. 시트의 '사람확인' 칸이 비어 있는 것도 사람이 검토하라는 안전장치입니다.

### Q3. 루프마다 세션을 따로 열어야 하나요?

아니요. 이 가이드는 **하나의 프로젝트 세션**에서 `LOOP.md`에 등록된 루프들을 함께 실행하는 방식입니다. 세션 이름이 아니라 `LOOP.md`의 루프 이름과 실행 프롬프트가 기준입니다.

### Q4. `/loop`과 `/goal`, Hermes cron은 뭐가 다른가요?

`/loop`은 간격마다 반복(내 PC·세션·최대 7일), `/goal`은 정한 조건이 될 때까지 계속, Hermes cron은 정해진 시간에 영속적으로 일하고 보고하는 직원형입니다. 위 [비교표](#loop--goal--hermes-cron-구분)를 참고하세요.

### Q5. 비용(토큰)이 많이 들까요?

루프가 반복 실행될 때마다 구독 사용량(quota)을 씁니다. 처음부터 많이 돌리지 말고 **2~3개부터** 시작하는 걸 권합니다.
